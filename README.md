# OnlineCall

Локальное приложение для созвонов. Оно запускается на вашем ПК, само раздает сайт и держит WebSocket signaling на `/ws`. Наружу его можно открыть через любой HTTPS reverse tunnel: CloudPub, Tuna, xTunnel, Tunyl, Cloudflare Tunnel, ngrok или аналог.

## Что умеет

- Свой локальный Bun-сервер без GitHub Pages.
- Встроенный WebSocket signaling, без публичного PeerJS.
- Короткий код комнаты и ссылка-приглашение.
- Вход без камеры и микрофона.
- Аудио, видео, выбор устройств.
- Демонстрация экрана.
- Чат через signaling-сервер, поэтому он работает сразу после входа.
- Автопереподключение signaling.
- WebRTC perfect negotiation для устойчивой смены медиа/экрана.
- Базовое встроенное шифрование WebRTC: DTLS/SRTP для медиа.

## Быстрый запуск

Нужен Bun 1.3+.

```bash
bun install
bun run build
bun run start
```

По умолчанию сервер слушает `0.0.0.0:3000`.

Локально можно открыть:

```text
http://localhost:3000
```

Для созвона через интернет прокиньте порт `3000` через HTTPS-туннель. В туннельном сервисе нужен target:

```text
http://127.0.0.1:3000
```

Оба участника открывают публичную HTTPS-ссылку туннеля.

## Требования к туннелю

Сервис должен поддерживать:

- публичный `https://...` URL;
- WebSocket proxying для `/ws`;
- достаточно длинную сессию для вашего звонка.

Из перечисленных вариантов сначала пробуйте `CloudPub`, потому что для длинных сессий он выглядит самым подходящим. `Tuna` годится для коротких тестов из-за лимита 30 минут. `xTunnel` может упереться в лимиты запросов/трафика. `Tunyl` можно пробовать, если он нормально держит WebSocket.

## Как пользоваться

1. Хост запускает `bun run start`.
2. Хост запускает туннель на локальный порт `3000`.
3. Хост открывает HTTPS-ссылку туннеля.
4. Хост копирует `Ссылка` или `Код`.
5. Второй участник открывает эту же HTTPS-ссылку или вводит код комнаты.
6. Камеру и микрофон можно включить сразу или позже.

Статус должен быть `локальная комната открыта` или `Вы в локальной комнате`. Это значит, что используется ваш локальный signaling.

## Важное ограничение

Туннель заменяет signaling, но не является TURN-сервером. Медиа идут через WebRTC напрямую между браузерами. У большинства домашних сетей это работает через STUN. Если у кого-то строгий NAT или корпоративная сеть, может понадобиться TURN-сервер.

## Проверка

Сначала:

```bash
bun run build
bun run preview
```

В другом терминале:

```bash
bun run test:e2e:all
```

Проверка, что приложение работает без публичного PeerJS:

```bash
E2E_BROWSER=chromium E2E_BLOCK_PEERJS=true E2E_SKIP_MEDIA=true bun run scripts/e2e-webrtc.ts
E2E_BROWSER=chromium E2E_BLOCK_PEERJS=true bun run scripts/e2e-webrtc.ts
```
