# OnlineCall

Статическое приложение для созвонов на GitHub Pages. Участник открывает сайт, получает короткий код комнаты или ссылку, второй участник входит по этому коду. Камера и микрофон не обязательны: можно подключиться только для чата или просмотра.

## Что умеет

- Next.js static export для GitHub Pages.
- Автоматическое подключение по короткому коду комнаты.
- Ссылка-приглашение с `?room=...`.
- Вход без камеры и микрофона.
- Аудио, видео и переключение устройств.
- Mute / camera off.
- Демонстрация экрана.
- Чат между участниками.
- Список участников и статус подключения.
- PWA-манифест и service worker.
- Базовое встроенное шифрование WebRTC: DTLS/SRTP для медиа и SCTP/DTLS для DataChannel.

## Как пользоваться

1. Первый участник открывает сайт. Комната создается автоматически.
2. Он копирует `Ссылка` или `Код`.
3. Второй участник открывает ссылку или вставляет код в поле `Войти по коду`.
4. После подключения можно включить камеру/микрофон, чат или демонстрацию экрана.

Код комнаты короткий, обычно 6 символов. Полная ссылка длиннее, потому что в ней есть адрес GitHub Pages.

## Важное ограничение

GitHub Pages умеет отдавать только статические файлы. Он не может сам держать WebSocket, API, комнаты и очередь сообщений. Поэтому для автоматического входа используется бесплатный публичный PeerJS signaling. Он нужен только для первичного поиска участников; медиа и чат идут через WebRTC.

Если публичный signaling временно недоступен или блокируется сетью/браузером, приложение не падает в тупик: оно продолжает переподключаться и показывает кнопку `Повтор`.

TURN-сервера в бесплатной GitHub Pages-only схеме нет. У большинства домашних сетей WebRTC работает через STUN, но у строгих NAT или корпоративных сетей прямое соединение может не установиться. Для полностью надежных звонков через любые сети нужен отдельный TURN/signaling-сервер.

## Tunnel / LAN режим

Если публичный PeerJS или Tailscale заблокирован, можно запустить приложение локально и открыть его наружу через любой HTTPS reverse tunnel: CloudPub, Tuna, xTunnel, Tunyl, Cloudflare Tunnel, ngrok или аналог.

В этом режиме локальный Bun-сервер делает две вещи:

- отдает статический сайт из `out`;
- держит локальный WebSocket signaling на `/ws`.

Запуск на компьютере-хосте:

```bash
bun install
bun run build
bun run tunnel
```

После этого туннельным сервисом нужно прокинуть локальный порт `3000` наружу. Сервис должен поддерживать:

- публичный `https://...` URL;
- WebSocket proxying для пути `/ws`;
- достаточно длинную сессию для созвона.

Оба участника открывают именно HTTPS-ссылку туннеля. Камера и микрофон в браузере надежно работают только в secure context: `https://...` или `localhost`.

Туннель заменяет публичный PeerJS signaling, но не является TURN-сервером. Если WebRTC не сможет соединить участников напрямую через STUN, для таких сетей все равно нужен TURN.

## Локальная разработка

Нужен Bun 1.3+.

```bash
bun install
bun dev
```

Открыть:

```text
http://localhost:3000
```

## Проверка перед деплоем

```bash
bun run check
bun run audit
bun run preview
```

После `bun run build` статический сайт лежит в папке `out`.

`bun run preview` запускает статический сервер из папки `out`. Если порт `3000` занят, preview автоматически попробует следующие порты и напечатает фактический URL.

## E2E-проверка звонка

Тест открывает две независимые вкладки, создает комнату, подключает второго участника, проверяет участников, чат, режим без медиа и демонстрацию экрана в Chrome.

Сначала запустить preview:

```bash
bun run preview
```

В другом терминале:

```bash
bun run test:e2e:all
```

Проверяются:

- Chrome: вход по введенному коду.
- Chrome: вход по общей ссылке комнаты.
- Chrome: вход без камеры и микрофона.
- Firefox: вход по введенному коду.
- Firefox: вход без камеры и микрофона.
- Чат после подключения.
- Chrome: включение и выключение демонстрации экрана.

Проверка, что локальный signaling работает без публичного PeerJS:

```bash
E2E_BROWSER=chromium E2E_BLOCK_PEERJS=true E2E_SKIP_MEDIA=true bun run scripts/e2e-webrtc.ts
E2E_BROWSER=chromium E2E_BLOCK_PEERJS=true bun run scripts/e2e-webrtc.ts
```

## Деплой на GitHub Pages

1. В репозитории открыть `Settings -> Pages`.
2. В `Build and deployment` выбрать `GitHub Actions`.
3. Запушить изменения в ветку `main`.
4. Workflow `.github/workflows/deploy.yml` сам соберет `out` и опубликует сайт.

Конфиг автоматически выставляет `basePath` под имя репозитория, поэтому проектные GitHub Pages вида `https://user.github.io/repo/` работают без ручной правки.
