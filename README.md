# Manual Meet

Статическое приложение для созвонов на GitHub Pages. Серверной части нет: аудио, видео, демонстрация экрана и чат работают через WebRTC, а первичное подключение делается ручным обменом `offer` / `answer` кодами.

## Что умеет

- Next.js static export для GitHub Pages.
- WebRTC 1-на-1 как основной сценарий.
- Малые комнаты без сервера: нужно соединить каждую пару участников вручную.
- Аудио и видео.
- Выбор камеры и микрофона.
- Подключение без камеры и микрофона: можно войти как слушатель или только для чата.
- Mute / camera off.
- Демонстрация экрана через `replaceTrack`, без повторного offer/answer.
- Чат через WebRTC DataChannel.
- Базовое шифрование WebRTC: DTLS/SRTP для медиа и SCTP/DTLS для DataChannel.
- PWA-манифест и service worker.

## Важное ограничение

GitHub Pages отдает только статические файлы. Он не умеет WebSocket, API, хранение комнат и пересылку сообщений между участниками. Поэтому автоматического входа по ссылке как в Zoom/Jitsi здесь нет.

Ссылка комнаты нужна для удобства и защиты от случайного совпадения, но участники все равно должны вручную обменяться кодами подключения в личном канале.

## Как пользоваться 1-на-1

1. Оба открывают сайт.
2. Камеру и микрофон можно включить сразу или позже. Для подключения они не обязательны.
3. Первый участник нажимает `Создать offer`.
4. Первый копирует код и отправляет второму.
5. Второй вставляет код в поле `Код от собеседника` и нажимает `Применить код`.
6. Второй копирует появившийся `answer` и отправляет первому.
7. Первый вставляет `answer` и нажимает `Применить код`.
8. После соединения работает видео, аудио, экран и чат.

Второму участнику не обязательно заранее открывать ту же ссылку комнаты. Если он открыл сайт отдельно и вставил `offer`, приложение автоматически переключит его комнату на комнату из `offer`, чтобы не было рассинхрона.

Карточка показывает 50-символьное короткое превью кода. Для реального подключения нужно копировать полный `offer` / `answer` из поля `Ваш код для собеседника` или кнопкой копирования рядом с ним. Без сервера полный WebRTC-код нельзя честно сократить до 50 символов: в нем лежат SDP, ICE, fingerprint и параметры защищенного соединения.

## Как пользоваться группой

Без signaling-сервера группа работает как ручная mesh-сеть. Каждый участник должен соединиться с каждым нужным собеседником отдельной карточкой подключения.

Практичный лимит для такого режима: 3-4 человека. До 10 технически возможно, но вручную неудобно и нагрузка на интернет/CPU будет высокой.

## Локальная разработка

Нужен Bun 1.3+.

```bash
bun install
bun dev
```

Открыть:

```text
http://localhost:3000
```

## Проверка перед деплоем

```bash
bun run check
bun run audit
bun run preview
```

После `bun run build` статический сайт лежит в папке `out`.

`bun run preview` запускает статический сервер с папки `out`. Если порт `3000` уже занят, preview автоматически попробует следующие порты и напечатает фактический URL.

## E2E-проверка звонка

В проекте есть браузерный smoke-test, который открывает две независимые вкладки, включает fake camera/mic, делает ручной обмен `offer` / `answer`, проверяет WebRTC-соединение и чат.

Сначала запустить preview:

```bash
bun run preview
```

В другом терминале:

```bash
bun run test:e2e:all
```

Проверяются:

- Chrome: участники открыли разные комнаты, второй принимает комнату из `offer`.
- Chrome: оба заранее открыли одну ссылку комнаты.
- Chrome: оба подключаются без камеры и микрофона, затем проверяется чат.
- Firefox: участники открыли разные комнаты, второй принимает комнату из `offer`.
- Firefox: оба подключаются без камеры и микрофона, затем проверяется чат.
- Восстановление после невалидного `offer` в той же карточке подключения.
- Чат после WebRTC-соединения.
- Chrome дополнительно проверяет включение и выключение демонстрации экрана. Firefox-сценарий проверяет звонок и чат; screen share в headless Firefox не используется как надежный автотест.

## Деплой на GitHub Pages

1. Создать репозиторий на GitHub.
2. Запушить проект в ветку `main`.
3. В настройках репозитория открыть `Settings -> Pages`.
4. В `Build and deployment` выбрать `GitHub Actions`.
5. Workflow `.github/workflows/deploy.yml` сам соберет `out` и опубликует сайт.

Конфиг автоматически выставляет `basePath` под имя репозитория, поэтому проектные GitHub Pages вида `https://user.github.io/repo/` работают без ручной правки.

## Сеть

По умолчанию включен публичный STUN от Google. Это не signaling-сервер и не передает ваш звонок через Google, а помогает браузерам найти свои сетевые адреса. Если участники в одной локальной сети, можно выбрать режим `Только локальная сеть`.

TURN-сервера нет. Если у кого-то строгий NAT или корпоративная сеть, соединение может не установиться. Бесплатного надежного TURN без внешнего сервера обычно не бывает.
